# Известные подводные камни

Проблемы, реально встреченные при запуске скрейпера на текущих источниках (по состоянию на 2026-05). Каждый пункт — это либо засада, в которую я уже наступил, либо то, на чём упадёт следующий, кто будет повторять эту работу.

## 1. basketball-reference за Cloudflare TLS-challenge

Простой `requests` с любым `User-Agent` получает HTTP 403 и страницу `<title>Just a moment...</title>` с токеном `__cf_chl_rt_tk`. Это активная JA3-проверка TLS-отпечатка, которую обычным HTTP-клиентом не пробить, **сколько бы заголовков ты ни накидывал** (`Sec-Fetch-*`, `Referer`, реалистичный UA — не помогают). R-пакет `polite` падает по той же причине: 18 из 18 запросов возвращают `NULL`.

**Решение:** `curl_cffi.requests.Session(impersonate="chrome")` имитирует TLS-handshake Chrome и проходит первый раз. Это не браузер, а HTTP-клиент — запрет на selenium/playwright не нарушается.

## 2. hoopshype делает гео-редирект, и большинство URL ведут на пустую страницу

Цепочка для EU IP: `hoopshype.com` → 301 → `www.hoopshype.com` → 302 → `eu.hoopshype.com`. На EU-домене **нет страниц по сезонам**: любой `/salaries/players/2023-2024/` молча резолвится в `/` (главная) и отдаёт только 5 «top contracts» вместо ~600 игроков. Заголовки `CF-IPCountry: US`, `X-Forwarded-For: 8.8.8.8`, `Accept-Language: en-US` редирект **не обходят**. `us.hoopshype.com` не существует. Без прокси/VPN живой сайт за пределами US бесполезен.

**Решение:** Wayback Machine. URL вида `https://web.archive.org/web/{YYYYMMDD}id_/https://hoopshype.com/salaries/players/{start}-{end}/` (суффикс `id_` отдаёт raw HTML без обёртки Wayback) возвращает **старую версию страницы** с классической `<table class="hh-salaries-ranking-table">`. У Wayback свои особенности: запросы медленные (нужен таймаут ~90с) и они сами банят при частоте — throttle 5с обязателен. Snapshot не всегда есть точно на запрошенную дату, Wayback берёт ближайший — для сезона 2024 я просил `20240601`, получил `20240710`. Это не баг, но если для какого-то сезона снимок очень далёкий по времени — данные могут быть устаревшими.

## 3. Bbref сменил формат данных, и старые id/маркеры больше не работают

- **`id` таблицы advanced — это `advanced`, а не `advanced_stats`.** В R-коде стояло `advanced_stats` — таблица с таким id не существует ни на одной странице, поэтому даже если бы Cloudflare пропустил, парсер всё равно вернул бы пусто.
- **`TOT` → `2TM` / `3TM`.** Раньше для игроков, поменявших команду посреди сезона, итоговая строка имела `Tm = TOT`. После ~2022 формат поменялся на `2TM`/`3TM`/`4TM` (по числу команд). Старый фильтр `Tm == "TOT"` оставит дубли по командам и потеряет суммарную строку. Я фильтрую по `{TOT, 2TM, 3TM, 4TM}` сразу.
- **Slug игрока (`Player-additional`) теперь лежит как `<td data-append-csv="doncilu01">` внутри `<td data-stat="name_display">`**, и из ячейки нужно вытащить либо атрибут, либо `href` ссылки на `/players/d/doncilu01.html`. `pd.read_html` slug теряет — поэтому везде BeautifulSoup.

## 4. Формат All-Star меняется год от года

- 2016, 2024: id таблиц = `East` / `West`.
- 2018, 2019, 2020: id таблиц = `Team LeBron` / `Team Stephen` / `Team Giannis` (era капитанов).
- 2017, 2021–2023: смешанные варианты.

Парсер с захардкоженными id (`East`, `West`) сломается на середине диапазона. Я беру **все таблицы на странице кроме `id="line_score"`** — этот id стабилен, а остальные таблицы и есть ростеры.

## 5. Драфты 2022, 2023, 2024 содержат 58 пиков, а не 60

Это **не пропуск парсинга** — реальное состояние данных. Philadelphia 76ers лишились нескольких пиков за tampering (Daryl Morey + James Harden), плюс Milwaukee и Miami потеряли по одному. Если в санити-чеке проверять `assert n == 60` — упадёт. У нас фильтр на `dropna(subset=["player_id"])` сохраняет всё что есть, и в итоговом `draft_picks.csv` за эти годы будет 58 строк.

## 6. Канонизация имён теряет суффиксы — это и преимущество, и риск

`canonicalize_name()` намеренно срезает `Jr./Sr./II/III/IV` (на разных сайтах их пишут непоследовательно — `Kelly Oubre Jr.` на bbref, `Kelly Oubre` на hoopshype), и убирает диакритику + точки + апострофы. Это даёт чистый join по `(player_clean, season)`.

**Но:** «Glenn Robinson III» → `glenn robinson`. Если в датасете окажется и его отец Glenn Robinson (играл в 1994–2005), их `player_clean` совпадут. Для диапазона 2016–2024 конфликта нет, но если расширять назад — нужны алиасы в `data/lookups/name_aliases.csv` (шаблон уже создан, пустой). Список других известных коллизий проверяй вручную по логам `logs/unmatched_stats_without_salary.csv` после первого прогона.

## 7. ~16% игроков не имеют draft_pick — это норма, а не пропуск

В финальном `data_merged.csv` 612 из 3660 строк (16.7%) имеют `draft_year = NaN` — это **undrafted free agents** (Fred VanVleet, Wesley Matthews и т.п.). Колонка `undrafted = 1` для них уже посчитана. **Не пытайся** их матчить через расширение диапазона драфтов или скрейп каждой персональной страницы — это дёшево исправит только часть случаев, а основная масса реально вне драфта.

В регрессии обычно используют либо `undrafted` как dummy + `draft_pick` только для drafted, либо `draft_pick = 61` (псевдо-«61-й пик») для undrafted с осторожностью. У `pick_overall` тяжёлый хвост, поэтому log(draft_pick) или сама dummy надёжнее.

## Что НЕ является подводным камнем, хотя выглядит как он

- **«21% advanced без per_game»** в логе anti-join — by design. `per_game` отфильтрован по `GP>=20 & MPG>=5`, а `advanced` сохранён целиком. После финального merge всё равно фильтруется через `gp>=20`. То же про «23% salary без stats» — это игроки на 10-day и two-way контрактах с малым MPG.
- **128 NaN в `fg3_pct`** — игроки, не делавшие ни одного 3-очкового за сезон (классические центры). Деление на ноль; pandas корректно даёт NaN.
- **Wayback snapshot 2024 датирован июлем 2024** — это нормально, контракты к лету уже финализированы и не меняются.
